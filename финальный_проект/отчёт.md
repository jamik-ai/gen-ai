# Отчёт: триаж IT-тикетов поддержки (классификация + RAG-ответ)

Трек B (прикладной).

## 1. Задача и данные

Задача: автоматический триаж тикетов IT-поддержки — определить категорию обращения, найти релевантные
правила/SLA во внутренней базе знаний и составить ответ клиенту, опираясь только на найденные факты (без
выдумывания цифр), плюс независимая проверка качества ответа.

Данные: `input/raw_subset_tickets.csv` — 3000 тикетов, 5 сбалансированных категорий `CategoryA..E` (по 600
каждая). Текст лемматизирован и анонимизирован: нет имён, почти нет дат, у части тикетов почти не осталось
содержательных слов (например `"client hi client currently managing thanks head mob"`).

Реального описания категорий в датасете нет. `scripts/prepare_data.py` восстановил вероятный смысл категорий
по частотному анализу отличительных слов (`output/category_keywords.json`):

| Категория | Рабочее название (наша гипотеза) | Ключевые слова |
|---|---|---|
| A | Hardware & Server Infrastructure | server, disk, test, machine, host, connect |
| B | Building Access — Visitor & Cards | card, visitor, floor, leaver, maternity |
| C | Building Access — Lost Cards & Incidents | card, lost, badge, door, manager |
| D | User Administration & Mailbox | user, change, interface, mailbox, leave |
| E | Procurement & Equipment Purchase | item, laptop, purchase, order, allocation |

Это гипотеза, не подтверждённый факт — B и C почти неразличимы по словарю (обе про карты доступа), что и
подтвердилось в eval (см. п.4). На основе этой гипотезы вручную написана **синтетическая** база знаний
`kb/` (15 markdown-документов, 3 на категорию, с конкретными SLA/суммами/контактами) — реальной KB для этого
датасета не существует, это вынужденное допущение прикладного трека.

Eval-сет: стратифицированная выборка 30 тикетов (6 на категорию) из `input/test_tickets.csv`.

## 2. Система

Мультиагентный пайплайн без фреймворков (свой Python-код, по образцу паттернов курса):

```
тикет → ClassifierAgent (IE, structured output) → category, urgency, key_topic, SLA-оценка
      → ResponderAgent (tool-calling агент: сам вызывает search_kb по BM25-индексу KB) → ответ + cited_facts
      → JudgeAgent (LLM-as-judge: вердикт по каждому факту + overall_score)
      → если overall_score < 0.6 → один rework: Responder переписывает ответ с фидбеком судьи
      → программная проверка галлюцинаций (ghost-числа/факты против retrieved KB-чанков)
```

Техники курса, которые реально работают: **RAG** (BM25 search_kb), **агент с инструментами** (Responder сам
решает, когда и что искать), **мультиагент** (Classifier/Responder/Judge со structured handoff и
rework-циклом), **LLM-as-judge** (Judge, температура 0.7 — иначе судья просто соглашается с моделью).
Отдельно: structured output с `field_validator` (`TicketExtraction.estimated_resolution_days`, бизнес-
инвариант 0-30 дней) и проверка галлюцинаций (`hallucination.py`).

Модель: `openai/gpt-4o-mini` через OpenRouter (почему не DeepSeek напрямую — см. п.5).

## 3. Оценка

Eval на 30 тикетах (`output/eval_table.csv`, `output/eval_summary.json`):

| Метрика | Значение |
|---|---|
| Classification accuracy | **53.3%** (16/30) |
| Judge pass-rate (overall_score ≥ 0.6) | 43.3% |
| Средний overall_score судьи | 0.48 |
| Category consistency (по мнению судьи) | 70.0% |
| Rework rate | 63.3% |
| Ghost numbers (выдуманные цифры) | **0 / 29** числовых утверждений |
| Ghost cited facts | **0 / 19** проверяемых фактов |
| Тикетов без релевантной KB (no_kb_retrieved) | 53.3% |
| LLM-вызовов / tool-вызовов на тикет (среднее) | 3.27 / 3.80 |
| Стоимость прогона (30 тикетов) | $0.0132 (82864 prompt + 5840 completion токенов) |

Confusion matrix (gold → pred):

| gold | pred (распределение) | accuracy по категории |
|---|---|---|
| A | A:4, D:1, E:1 | 4/6 |
| B | B:4, A:1, D:1 | 4/6 |
| C | A:2, D:1, C:2, B:1 | 2/6 |
| D | D:6 | **6/6** |
| E | A:4, D:1, C:1 | **0/6** |

Путь (path): в среднем 3.27 LLM-вызова и 3.8 tool-вызова на тикет — это classify (1) + responder (1-2 шага
tool-loop) + judge (1) + при rework (63% случаев) ещё один цикл responder+judge. 0% ghost-rate — не артефакт
измерения (проверено вручную, см. trace TCK-0007): когда KB найдена, gpt-4o-mini точно опирается на неё; когда
не найдена — модель честно пишет "не нашёл, эскалирую", а не выдумывает цифру. Системный промпт ("числа
никогда не придумывай") в связке с RAG сработал.

## 4. Где пайплайн ошибался

**1) CategoryB ↔ CategoryC — неразличимый словарь.** Тикет `TCK-0016`: *"sent thursday re card se la sent
thursday december card va sa se va la data analyst ext hub"* (gold=C, "инцидент с картой"). Классификатор
выдал B (conf=0.8, "выпуск карты") — обе категории про карты доступа, а текст обезличен до набора шумовых
токенов. Responder, ограниченный категорией B, нашёл только документы про visitor/leaver/maternity и честно
ответил "не нашёл информацию, эскалирую". Судья оценил 0.85 (категория внутренне согласована, но полезности
ноль) — ошибка классификации каскадом убила retrieval, хотя нужный факт (стоимость замены карты, $15, SLA 2
дня) лежал в `C1_lost_card.md`, в соседней категории.

**2) CategoryE → CategoryA: похоже, проблема не в модели, а в самой разметке.** Тикет `TCK-0028`: *"...got
laptop for months already intern spotted there specific period when computer disk memory loaded by processes
task manager performance shows freezes..."* (gold=E "Purchase"). Модель классифицировала как A
("Hardware/Infrastructure", conf=0.8) — и по содержанию это куда логичнее: тикет описывает зависания ноутбука
из-за нагрузки на диск, а не запрос на покупку. **CategoryE целиком провалена в этом eval-сете — 0 из 6**, и
почти все её тикеты (4/6) ушли в A. Похоже, в исходной разметке "Purchase" — это организационный маршрут
(IT-asset team отвечает и за закупку, и за железо принадлежащих компании ноутбуков), а не содержательная
категория, различимая по тексту тикета. Это системная, не случайная ошибка.

**3) Деградация текста ломает retrieval даже при верной категории.** Тикет `TCK-0022`: *"client hi client
currently managing thanks head mob"* (gold=D, предсказано верно D, conf=0.7). Но в этом обрубке текста нет ни
одного слова, пересекающегося с тремя KB-статьями категории D (user rights / mailbox handover / report
access) — после анонимизации тикет содержит 7 слов, из них 5 — приветственная "вода". Responder корректно
сообщил об отсутствии информации, но судья поставил 0.25 и `category_consistent=false` — по его мнению,
ответ "про управление клиентами" не похож на User Administration. Здесь расходятся три источника истины:
gold-метка (D), решение классификатора (D, верно) и субъективное суждение judge (несогласен) — у самого
judge нет доступа к gold-метке, и его оценка местами строже, чем фактическая правильность пайплайна.

## 5. Выводы

**Ответ на исходный вопрос** — можно ли собрать рабочий, не выдумывающий цифры триаж-пайплайн на сильно
обезличенных тикетах с синтетической KB: да, по hallucination-метрике (0% ghost-rate) пайплайн надёжен —
система предпочитает честно сказать "не знаю", а не придумать SLA. Но по полезности (accuracy 53%, judge
pass-rate 43%) система на сегодняшнем уровне — черновик, не production-ready: больше половины тикетов либо
классифицируются неверно, либо не находят релевантную KB, либо и то, и другое одновременно.

**Главное узкое место — retrieval, а не генерация.** В процессе нашли и исправили дизайн-ошибку: KB была
написана на русском (по конвенции курса), а тикеты/ответы — на английском, из-за чего BM25 почти не находил
пересечений токенов; переписали 15 документов на английский, что заметно подняло качество retrieval. Но даже
после фикса `no_kb_retrieved` осталась на 53.3% — теперь по двум остальным причинам: (а) каскад ошибок
классификации (9 из 16 случаев — поиск идёт в неправильной категории, см. п.4.1); (б) узкое покрытие
синтетической KB (3 документа на категорию не покрывают всё разнообразие реальных подтем) в сочетании с
экстремальной деградацией текста после анонимизации (см. п.4.3).

**Что оставили бы в прод:** мультиагентную структуру Classifier→Responder→Judge с rework-циклом и
программную проверку галлюцинаций по цифрам (язык-независимая, надёжная). **Что переделали бы:** (1) BM25 →
гибридный dense+sparse retrieval (как в семинаре 4) — устойчивее к парафразам и коротким зашумлённым запросам;
(2) расширили бы KB кратно (десятки документов на категорию, а не 3); (3) сделали бы classification и retrieval
менее жёстко связанными — например, дать Responder возможность искать по top-2 категориям классификатора, а не
только по top-1, чтобы ошибка классификации не гарантированно убивала retrieval; (4) для CategoryE отдельно
проверили бы с экспертом, не является ли это организационным, а не содержательным признаком — текстовый
классификатор такое в принципе не выучит.

**Границы применимости:** оценка получена на одной модели (gpt-4o-mini) и одном случайном стратифицированном
сэмпле из 30 тикетов — выводы про конкретные confusion-паттерны (особенно "CategoryE = 0%") нужно
перепроверить на большей выборке перед тем, как делать организационные выводы. Подпись категорий — наша
гипотеза по словам, не подтверждённая владельцем датасета; реальная KB для этого продукта не существует, вся
KB синтетическая и написана нами.

### Технические заметки прогона (для воспроизводимости)

- DeepSeek-аккаунт (использовавшийся в курсе ранее) исчерпал баланс на 15-м из 30 тикетов в процессе работы —
  переключились на OpenRouter (`openai/gpt-4o-mini`). При этом `deepseek/deepseek-chat` через OpenRouter
  отдавал битый вывод (литералы tool-call токенов как текст вместо структурированных tool_calls) — для
  tool-calling агента это критично, поэтому ушли на `gpt-4o-mini`.
- Найден и исправлен баг в унаследованном `llm_client.py` (общий для всех семинаров курса): внутренняя
  функция `_call(kw: dict)` затеняла внешний `**kw` (куда передаётся, например, `max_tokens`) — параметр
  молча терялся для всех structured-вызовов через `response_model=`.
