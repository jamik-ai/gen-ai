**Autodiscovery:**

Модель переоткрыла категории из IE-схемы, но с немного другой гранулярностью (разделила `data_loss` и `data_integrity`).

## 2. Где модель ошибалась

**Пример 1 — ошибка в аспекте `scope`, тикет T009:**  
Текст: «Перетаскиваю задачи в Kanban — они становятся на нужные места, но после обновления страницы всё возвращается в прежний порядок. **Команда тратит время на постоянную переорганизацию доски.**»  
Модель поставила `scope = positive` (узкий охват), использовав цитату «Команда тратит время» — при том что слово «команда» прямо указывает на нескольких пользователей. Правильный ответ: `scope = negative`.

**Пример 2 — ошибка в аспекте `urgency`, тикет T004:**  
Текст: «С карты списывают деньги дважды... подписка за октябрь списана дважды — 5 октября и 7 октября.»  
Модель поставила `urgency = neutral`, опираясь на цитату «Прошу вернуть один платёж» (вежливая просьба). Финансовое двойное списание — по смыслу срочная проблема (`urgency = negative`), но модель ориентировалась на тон, а не на категорию проблемы.

**Пример 3 — пограничный случай у судьи, рекомендация про документацию:**  
Рекомендация: «Обновить документацию по SSO и API, добавить информативные сообщения об ошибках.»  
Судья выдал `supported`, указав на три тикета: T015 (устаревшая документация SSO), T011 (API возвращает 401 без пояснений), T020 (ошибка без объяснения при превышении лимита).  
Однако T011 — это баг авторизации, а не проблема документации; включение его как подтверждения «обновить документацию» — спорно. Строгий судья мог бы поставить `weakly_supported`: только 1 тикет (T015) напрямую жалуется на документацию, остальные два — об информативности ошибок, что является отдельной задачей UX.

**Autodiscovery vs. фиксированные аспекты:**  
Фиксированный список описывает измерения тикета, динамический — содержательные категории. Интересно, что модель обнаружила `security_access` как самостоятельный аспект (T026 — утечка чужих данных, T034 — блокировка 2FA), чего в фиксированном списке нет совсем.

## 3. Что бы изменили в production

**Оставили бы как есть:**
- `llm_client.py` с JSON-mode + retry: ни одного необработанного падения за 23 вызова
- Пакетная обработка по 8 тикетов: надёжный баланс контекст / стабильность
- `check_quotes` по первым 30 символам: простая эвристика, 0% ложных срабатываний

**Переделали бы:**
- **Схему аспектов**: двухуровневая структура — мета-аспекты (severity/urgency/scope/resolution) для приоритизации + динамические категории для роутинга тикета к нужной команде
- **Разбиение**: сейчас фиксированное по числу тикетов (8 штук), в production — по числу токенов, иначе один тикет с длинным текстом может превысить лимит
- **Промпт аспектного анализа**: добавить явное правило «`scope = negative` если затронуто 2+ человека» — модель ошибается именно на этом граничном случае

**Добавили бы:**
- **Кэш ответов**: при итерациях над промптами повторный прогон на тех же тикетах тратит токены впустую
- **A/B на двух моделях**: сравнить IE-качество DeepSeek и GPT-4o-mini, выбрать лучшую по доле правильных `scope`
- **Агрегированный отчёт по тарифам**: enterprise-тикеты стабильно критичнее (T003, T015, T026, T040) — это важно для приоритизации очереди поддержки
